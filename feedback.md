# Code Review: IELTS Preparation App (Backend)

## 📊 Общая оценка

**Оценка: 6/10**

Проект имеет хорошую базовую структуру, модульную архитектуру и использует современные инструменты (NestJS, Drizzle ORM, BullMQ). Однако присутствуют критические проблемы в бизнес-логике, механизме авторизации и отдельных архитектурных решениях, которые необходимо исправить перед дальнейшим масштабированием.

---

## 💪 Сильные стороны

### Архитектура

- Хорошее разделение приложения на модули в соответствии с принципами NestJS:
  - Auth
  - Users
  - Exams
  - Attempts
- Четкое разграничение ответственности между слоями приложения.

### База данных

- Аккуратно описанные схемы Drizzle ORM.
- Корректно настроены связи между таблицами.
- Используется каскадное удаление (`onDelete: 'cascade'`).
- Применяется составной уникальный индекс для пользовательских ответов, что предотвращает дублирование данных.

### Транзакции

- Грамотное использование транзакций при:
  - создании полного экзамена (`createCompleteExam`);
  - сохранении попыток (`AttemptsRepo.update`).
- Исключается частичная запись данных при возникновении ошибок.

### Очереди (BullMQ)

- Проверка письменных ответов вынесена в отдельный Worker (`AttemptProcessor`).
- Тяжелые AI-операции не блокируют основной поток приложения.

### Безопасность

- Пароли пользователей хэшируются через bcrypt с использованием соли.
- Отсутствует хранение паролей в открытом виде.

## ⚠️ Слабые стороны и уязвимости

### JWT-аутентификация

#### Отсутствует срок жизни Access Token

В методе `generateTokens()` access token подписывается без параметра `expiresIn`.

Последствия:

- токен действует бессрочно;
- повышается риск компрометации аккаунтов.

#### Некорректная работа Refresh Token

Refresh Token не сохраняется в базе данных или Redis.

В результате:

- невозможно реализовать полноценный logout;
- невозможно отозвать скомпрометированный токен;
- проверка refresh token фактически не работает.

---

### AI Service

#### Хардкод модели

В `AiService` реализован список резервных моделей (`modelFallBackList`), однако внутри `executeRequest()` используется жестко заданная модель:

```ts
model: 'nvidia/nemotron-3-nano-omni-30b-a3b-reasoning:free'
```

Следствие:

- механизм fallback никогда не используется;
- переключение между моделями не работает.

---

### Логика оценки Writing

В IELTS Writing состоит из двух независимых заданий:

- Task 1
- Task 2

Сейчас Worker:

1. Склеивает все ответы в один текст (`combinedText`).
2. Использует тему только первого вопроса:

```ts
writingAnswers[0]?.questionId
```

Проблемы:

- Task 1 и Task 2 оцениваются как один текст;
- теряется точность оценки;
- не вычисляется общий балл (`overall score`).

---

### DTO и ValidationPipe

В `AttemptCreateDto` поле:

```ts
@IsNumber()
userId: number;
```

обязательное.

Однако в контроллере оно подставляется из JWT:

```ts
attempt.userId = req.user.sub;
```

При включенном глобальном ValidationPipe запрос без `userId` будет отклонен еще до выполнения контроллера.

Результат:

- пользователь получает `400 Bad Request`;
- логика контроллера не достигается.

---

### Типизация и схема БД

#### Использование `any`

В `ExamsRepository`:

```ts
sections: any[]
```

Использование `any` снижает надежность TypeScript и скрывает ошибки на этапе компиляции.

#### Использование `serial`

Во многих таблицах применяются инкрементальные идентификаторы:

```sql
serial
```

Недостатки:

- раскрывается количество пользователей;
- раскрывается количество экзаменов;
- облегчается перечисление ресурсов через API.

Для публичных сущностей предпочтительнее использовать UUID.

#### Отсутствие временных меток

Во многих сущностях отсутствуют поля:

```ts
createdAt
updatedAt
```

Это усложняет:

- аудит;
- отладку;
- аналитику;
- отслеживание изменений.

---

### Избыточный Exception Filter

Кастомный `DomainExceptionFilter` перехватывает:

```ts
NotFoundException
```

и возвращает практически тот же JSON, который NestJS генерирует по умолчанию.

Следствие:

- дополнительный код без практической пользы;
- усложнение поддержки проекта.

## 🛠 Рекомендации

### Шаг 1. Исправить AI fallback

Передавать модель параметром в `executeRequest()` вместо использования хардкода.

Пример:

```ts
private async executeRequest(model: string) {
    return this.client.chat.send({
        model
    });
}
```

---

### Шаг 2. Исправить Refresh Token Flow

При авторизации:

1. генерировать refresh token;
2. сохранять его в БД или Redis.

При обновлении токена:

1. сверять токен с сохраненным значением;
2. выдавать новую пару токенов только после успешной проверки.

---

### Шаг 3. Переработать оценку Writing

Для каждого ответа:

```ts
for (const answer of writingAnswers)
```

необходимо:

1. получать тему через:

```ts
getWritingTopic(answer.questionId)
```

2. отдельно отправлять текст в AI;
3. получать отдельный балл для каждого задания.

После этого вычислять:

- Writing Score;
- Overall Band Score.

---

### Шаг 4. Убрать `userId` из DTO

Вместо передачи через запрос:

```ts
POST /attempts
```

получать пользователя исключительно из JWT.

Например:

```ts
startAttempt(userId: number, dto: AttemptCreateDto)
```

---

### Шаг 5. Усилить типизацию

Заменить:

```ts
any[]
```

на:

- DTO;
- интерфейсы;
- типизированные структуры данных.

Также рекомендуется использовать:

```ts
@nestjs/mapped-types
```

для наследования DTO.

---

### Шаг 6. Рассмотреть переход на UUID

Для публичных сущностей:

- users
- exams
- attempts

использовать:

```sql
uuid
```

вместо:

```sql
serial
```

## 🚀 Направления для масштабирования

### Rate Limiting

Использовать уже подключенный Redis для ограничения запросов к AI-сервису.

Это позволит:

- предотвратить злоупотребления;
- защитить лимиты OpenRouter;
- снизить расходы.

---

### Bull Board

В проекте уже подключен `BullBoardModule`, однако очереди не зарегистрированы.

Рекомендуется добавить:

```ts
BullMQAdapter
```

для очереди:

```ts
exam-checking
```

чтобы получить интерфейс мониторинга:

```text
/admin/queues
```

Возможности:

- просмотр активных задач;
- просмотр упавших задач;
- повторный запуск задач;
- мониторинг производительности.

---

### Индексация базы данных

PostgreSQL не создает индексы для Foreign Key автоматически.

Рекомендуется добавить индексы для:

```sql
user_id
exam_id
attempt_id
question_id
```

Это существенно ускорит запросы при росте объема данных.

---

### Transactional Outbox Pattern

При увеличении нагрузки стоит рассмотреть внедрение Transactional Outbox.

Преимущества:

- отсутствие потери задач при перезапуске Redis;
- гарантированная доставка событий;
- согласованность между БД и очередями.

Сценарий:

1. Сохранение попытки в БД.
2. Сохранение события в Outbox в той же транзакции.
3. Фоновый процесс переносит событие в BullMQ.

---

## Итог

Проект уже имеет крепкий фундамент благодаря хорошей модульной архитектуре, использованию транзакций и асинхронной обработке задач через BullMQ. Основные проблемы сосредоточены в области безопасности JWT, логики оценки Writing и типизации. После устранения этих недостатков качество проекта может вырасти примерно до **8–8.5/10** и будет готово к дальнейшему масштабированию.
