# Code Review: IELTS Preparation App (Backend)

## 📊 Общая оценка
**Оценка: 6/10**
Проект имеет хорошую базовую структуру, модульную архитектуру и использует современные инструменты (NestJS, Drizzle ORM, очереди BullMQ). Однако, присутствуют критические ошибки в бизнес-логике (хардкод AI-модели, неправильное оценивание Writing части), уязвимости в механизме JWT-авторизации и избыточный/противоречивый код в слое контроллеров. 

## 💪 Сильные стороны (Strengths)
- **Архитектура**: Хорошее разделение приложения по модулям в духе NestJS (auth, users, exams, attempts).
- **База Данных**: Чисто описанные схемы Drizzle ORM. Хорошо продуманы связи таблиц (relations) и каскадное удаление (`onDelete: 'cascade'`). Уместное использование составного уникального индекса для ответов пользователя (`uniqueIndex`).
- **Транзакции**: Грамотное использование Drizzle транзакций при создании полного экзамена (`createCompleteExam`) и сохранении попыток (`AttemptsRepo.update`), что спасает от частичной записи.
- **Очереди (BullMQ)**: Вынесение тяжелых задач по оценке текстов AI в отдельный Worker (`AttemptProcessor`) — правильный архитектурный ход, не блокирующий event loop.
- **Безопасность**: Правильное использование bcrypt с солью для хэширования паролей пользователей.

## ⚠️ Слабые стороны и уязвимости (Weaknesses)
- **Уязвимости Auth (JWT)**: Метод `generateTokens` подписывает `access_token` без указания времени жизни (expiresIn). Рефреш-токены (`refresh_token`) никак не сохраняются в базе данных. Таким образом, привязка `refreshToken` не работает должным образом для инвалидации — реализовать безопасный логаут (logout) сейчас невозможно.
- **Хардкод AI-модели (Логическая ошибка)**: В `AiService` реализован массив fallback-моделей `modelFallBackList`, который перебирается в цикле в методе `analyzeText`. Однако, внутри функции `executeRequest` отправляемая модель захардкожена: `model: 'nvidia/nemotron-3-nano-omni-30b-a3b-reasoning:free'`. Резервные модели никогда не вызываются.
- **Слом логики оценивания (AttemptsProcessor)**: В IELTS есть 2 отдельных задания Writing (Task 1, Task 2). Сейчас Worker склеивает все письменные ответы в один текст (`combinedText`) и оценивает только по топику ПЕРВОГО вопроса (`writingAnswers[0]?.questionId`). Также в `scores` вообще не рассчитывается `overall` балл по всему экзамену.
- **Проблемы с ValidationPipe**: В `attemptCreateDto` поле `userId` задано как обязательное (`@IsNumber()`). В контроллере оно берется из JWT (`attempt.userId=req.user.sub`). Так как в NestJS включена глобальная валидация, клиентский запрос без `userId` отвалится с 400 Bad Request еще до того, как сработает код контроллера.
- **Схемы и Типизация**: В `ExamsRepository` метод `createCompleteExam` принимает аргумент `sections: any[]` (пропуск строгой типизации). В базе данных используются инкрементальные `serial` первичные ключи — это раскрывает бизнес-метрики (количество экзаменов/пользователей), лучше перейти на UUID. Нет полей `createdAt` и `updatedAt` у сущностей.
- **Фильтры Исключений**: Кастомный `DomainExceptionFilter` в `attempts/exceptions` перехватывает стандартный `NotFoundException` и возвращает тот же самый JSON, который NestJS вернул бы по умолчанию. Код избыточен.

## 🛠 Конкретные советы по улучшению (Recommendations)
- **Шаг 1**: В `AiService.executeRequest` замените захардкоженную модель на переменную `model`, передаваемую в функцию.
- **Шаг 2**: Добавьте сохранение сгенерированного `refresh_token` в таблицу `users` (или Redis) в момент авторизации. В методе `refreshToken` сверяйте токен из базы перед выдачей нового access-токена.
- **Шаг 3**: Перепишите цикл в `AttemptProcessor`, чтобы для каждого `writingAnswer` вызывался `this.examService.getWritingTopic(answer.questionId)` и отдельно происходил запрос в AI-сервис (с последующим вычислением среднего балла Writing и Overall балла).
- **Шаг 4**: Уберите поле `userId` из `attemptCreateDto` (или сделайте его `@IsOptional()`). Лучше передавать `userId` отдельным аргументом в `AttemptsService.startAttempt(userId, attempt)`.
- **Шаг 5**: Используйте `@nestjs/mapped-types` или явные интерфейсы (вместо `any[]`) для создания экзаменов. Смените тип `serial` на `uuid` для таблиц, доступных по публичным эндпоинтам.

## 🚀 Направление для масштабирования (Next Steps)
- **Rate Limiting**: Используйте уже подключенный Redis (и пакет `cache-manager`) для реализации жесткого Rate-Limiting по запросам к AI-ендпоинтам, чтобы предотвратить исчерпание лимитов OpenRouter.
- **Bull Board**: В `AppModule` импортирован `BullBoardModule`, но не добавлены очереди для отслеживания. Прокиньте `BullMQAdapter` для очереди `exam-checking`, чтобы появился UI дашборд (на `/admin/queues`) для мониторинга зависших или упавших AI-задач.
- **Оптимизация индексов БД**: Добавьте B-Tree индексы на внешние ключи (`user_id`, `exam_id` и т.д.), так как PostgreSQL не индексирует Foreign Keys автоматически. Это ускорит запросы при большом количестве попыток (Attempts).
- **Outbox Pattern**: В будущем, если сообщений к AI станет очень много, сохраняйте задачи на проверку прямо в БД в единой транзакции с сохранением попытки (Transactional Outbox), а уже оттуда перекладывайте в BullMQ, чтобы избежать потери данных при перезапуске брокера.
